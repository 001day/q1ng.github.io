Java反序列化初探2

q1ng
2025-12-19
java代码审计

上篇说到，TransformedMap利用。但在ysoserial中正统的CommonCollections1利用链用到的是LazyMap而不是TransformedMap

接下来就要好好分析分析LazyMap这个类是如何替换TransformedMap的。上文我们分析到，我们需要寻找调用transform的类

我们可以看到LazyMap.get()方法的逻辑，是不是有调用transform。尽管在sun.reflect.annotation.AnnotationInvocationHandler 的readObject方法中并没有直接调用到 Map的get方法，但AnnotationInvocationHandler类的invoke方法有调用到get

想要完成上述的操作，我们现在面临一个问题 — Java 动态代理

Proxy.newProxyInstance 的第一个参数是ClassLoader，我们用默认的即可，只要能加载出对象就行；第二个参数是我们需要代理的对象集合；第三个参数是一个实现了InvocationHandler接口的对象，里面包含了具体代理的逻 辑。

接下来用ai生成的例子来理解下

运行结果：

起初，proxyMap.get(“hello”)；调用了Map的get方法,实际上这里的代理会生成类似一个$Proxy0 类，是Map接口的实现，这个类的内部无论经过什么，最后都会 return this.handler.invoke() 。此时就进入了Handler类，在这个类中，最终输出运行结果。

我们回看 sun.reflect.annotation.AnnotationInvocationHandler ，会发现实际上这个类实际就 是一个InvocationHandler我们如果将这个对象用Proxy进行代理，那么在readObject的时候，只要 调用任意方法，就会进入到 AnnotationInvocationHandler#invoke 方法中，进而触发我们的 LazyMap#get。此时调用链就又打通了，我们重新生成一个poc

前面的分析都和之前相同，不同的从这里开始 Map lazyMap = LazyMap.decorate(innerMap, transformerChain);

我们现在还原下反序列化后，从AnnotationInvocationHandler.readObject()作为入口开始，for (Map.Entry<String, Object> memberValue : this.memberValues.entrySet()) ，调用entrySet()来遍历，此时的this.memberValues=proxyMap,这是一个代理，我们前文提到的，无论干什么，最后都会调用AnnotationInvocationHandler.invoke()，接下来就回到了AnnotationInvocationHandler这个类的invoke方法，前文也提到invoke方法中会调用get方法 InvocationHandler handler = (InvocationHandler) construct.newInstance(Override.class, lazyMap); 这步中传入lazyMap，即invoke方法里调用 this.memberValues.get()时，实际上调用的就是 lazyMap.get() 。现在进入到Object value = factory.transform(key); 此时的factory就是 transformerChain ，原因是Map lazyMap = LazyMap.decorate(innerMap, transformerChain);

现在调用链又一次完美闭合了！！
